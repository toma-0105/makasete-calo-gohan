## 1. サービス概要
「何を食べるか考えるストレス」をなくし、迷わず食事を続けられるようにするサービス。  
TDEE・アレルギー・食材条件をもとに献立を生成し、“考えなくても続く食生活”を実現します。

## 2. このアイデアはどこから生まれたか


### 2-1. きっかけとなった体験・感情
- いつ<br>
数ヶ月前、ダイエットに励んでいる友人と食事について話をしていた際です。

- どんな状況で<br>
友人が「毎日、1日の摂取カロリー内で献立を考えるのが一番のストレス。いっそ1週間分を誰かに決めてほしい」と話していました。

- どんな気持ちになったか<br>
その言葉を聞いた瞬間、強く共感しました。<br>私自身、過去にダイエットで挫折した際に一番の壁は「何を食べれば正解か」を判断し続けることが大きな負担になっていたからです。<br>迷うことが行動を止める原因であり、この“決める負担”を無くすことが継続の鍵だと感じ、この課題を解決したいと思いました。

### 2-2. なぜそれが「気になった」のか
- 自分の性格<br>
物事を正しく進めたいと考えるタイプで、曖昧な状態がストレスになりやすい。<br>
また、判断回数が多いと徐々に疲れてしまう傾向があります。

- 過去の経験<br>
自分自身や友人の経験から、「献立を考えること」自体が継続の障壁になっていると感じました。<br>
そのため、単に意志の問題ではなく、仕組みの問題であると考えるようになりました。


- 大切にしている価値観<br>
「無理に頑張ること」ではなく「頑張らなくても続く仕組み」に価値を感じています。<br>
継続できる状態を設計することが重要だと考えています。

## 3. 課題の整理

### 3-1. 表に見えている困りごと
1日の消費カロリー内で献立を考えることが負担であり、毎日「何を食べればよいのか？」を考え続けることがストレスとなっていることです。

### 3-2. 本当に解決したい課題は何か
ユーザーが「この食事で正しいのか」と悩み続けることによる不安と判断負担を取り除き、迷うことなく継続できる食生活を実現することです。

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー
 [生活・状況]<br>
基本的に自炊中心で、ダイエットや健康管理に関心はあるものの、毎日の食事内容を細かく考えることに負担を感じている方。

 [理由]<br>
仕事や日常生活に追われる中で、毎食ごとに献立やカロリーを考える必要があり、その負担が積み重なることで食事管理が継続しにくくなるため。

[環境設定]<br>
既存の食事記録・可視化アプリは利用しているものの、献立の決定やレシピ検索は依然として自分で行う必要があり、ツールを使いこなすこと自体に負担を感じている。

[理由]<br>
既存の食事管理アプリでは「記録」や「可視化」はできるものの、実際の食事内容を決めるまでには検索・比較・判断が必要になる。
その結果、献立を考える負担や入力の手間、「これで合っているのか」という不安が積み重なり、継続が難しくなる。  
本アプリでは「記録」ではなく「意思決定の代行」にフォーカスし、献立を自動生成することで判断回数を減らし、継続しやすい状態を実現する。

<br>

### 4-2. 自分とユーザーの距離
過去の自分および身近な人（友人）に該当します。  
自身の経験と共通の課題であるため、具体的な利用シーンを想定しやすいと考えています。

### 4-3. 実際に使われる可能性について
- まず最初に使ってくれそうな人は誰か？<br>
ダイエットを始めたばかりで、やる気はあるものの「具体的なメニュー選び」で負担を感じている方。

- その人は、どんな場面でこのアプリを思い出すか？<br>
毎日「今日の夕飯、○○カロリー内に収めるには何を作れば正解？」と、スーパーの売り場や帰宅中の電車内で途方に暮れたタイミング。

- 今の生活の中で、代わりに使っているものは何か？<br>
カロリー管理アプリのレシピ検索、SNS（Instagram/YouTube）のダイエット飯、または「結局いつもと同じ」コンビニのサラダチキン。

- それを置き換えてまで使う理由<br>
既存のカロリー管理アプリやインターネット検索では、レシピや情報は得られるものの、「何を食べるべきか」は自分で判断する必要があります。<br> 
そのため、毎回献立を考える負担が発生し、結果として継続が難しくなってしまいます。<br>  
一方で本サービスは、TDEEや食材条件をもとに献立を自動生成することで、「何を食べるか」を考える必要をなくし、ユーザーの意思決定の負担を大きく減らします。<br>  
その結果、これまで継続できなかった食事管理を無理なく続けられるようになるため、“考えること”が原因で続かなかった人にとって、行動までを自動化できる点に価値があります。

<br>

### 4-4. ユーザーがサービスを導入して利用するまでのイメージ(導入・継続の流れ）

 どんな場面・困りごとがきっかけで、このサービスを知る・思い出すか？

- ダイエット開始時、「何を食べればいいか分からない」と行き詰まったとき<br>
- 「カロリー管理が続かない」と挫折しかけたとき<br>
- 「もっと楽にダイエットを続けたい」と思ったとき<br>
<br>


**ユーザーは以下の流れでサービスを利用開始します。**

```
登録の流れ [合計所要時間 約9０〜120秒]

① TDEE診断(所要時間:約30秒)
入力項目:身長・体重・年齢・性別・活動レベル(例:週に1〜2回程度の運動) 1日の消費カロリーが算出される

② ユーザー登録 or ゲスト利用を選択(任意・所要時間:約30〜60秒)
ゲスト利用→登録不要で即座に献立を確認できる
ユーザー登録→献立の保存や履歴の確認が可能になる

③ アレルギーの有無(任意・所要時間:約30秒)※選択式

④ 「献立を生成」ボタンを押す

```

**ユーザーが得られる価値→入力が完了すると、即座に献立が表示される**<br>
- 「今日は何を食べよう」と悩む必要がなくなる<br>
- ゲストログイン利用で気軽に試せるため、「とりあえず使ってみる」ができる<br>

## 5. 既存サービス・競合調査
### 5-1. 似たサービスの調査

#### サービス名：あすけん<br>
URL https://www.asken.jp/
- どんなことができるか<br>
  - 食事記録・栄養分析・AIアドバイスを中心に、日々の食生活を管理<br>
  - 自分自身のダイエット日記をつけたり、他のユーザーの日記を閲覧<br>
  - 参考レシピの記載あり（あすけん公式 結局、これしか作らない！短いレシピ）<br>
- 本サービスとの違い
  - あすけんは、食事内容の記録と可視化に優れていますが、献立を自動生成する機能はありません。

<br>

#### サービス名：me:new（ミーニュー)<br>
URL https://menew.jp/
- どんなことができるか<br>
  - 最長1週間分の献立を作成できます<br>
  - 苦手な食材やアレルギー食材を避けた献立提案ができます<br>
  - 献立ごとのカロリー・塩分量の表示や、メモ機能もあります<br>
  - 買い物リストも自動で出るので、まとめ買いしやすいです<br>
- 本サービスとの違い<br>
  - me:newは、最長1週間分の献立を作成する機能を持っていますが、ユーザーごとの消費カロリー範囲内に合わせて献立を生成する機能はありません。<br>献立のカロリー表示は提供されますが、実際のカロリー計算や調整はユーザーが行う必要があります。

<br>

### 5-2. それでも自分が作りたい理由
#### 意思決定の負担が大きい<br>
既存サービスは食事の「記録」や「分析」には優れていますが、「次に何を食べるか」という最も重要かつ負担の大きい意思決定をユーザー自身に委ねている点に不満を感じました。

<br>

#### 情報はあるのに決められない<br>
ダイエットアプリを通じて多くの知識やレシピを得られても、個々の身体条件（TDEE）やスケジュールに最適化された「その日に何を食べればよいか」までは提示されません。<br>その結果、「結局何を食べれば正解なのか」が分からず迷ってしまい、自分でカロリーを計算する手間が継続の大きな障壁になっています。

<br>

#### 既存の自動生成ツールとのズレ<br>
献立生成に特化したサービス（me:new）も存在しますが、こちらは主に「家事の効率化」や「節約」を目的としています。<br>ダイエットにおいて最も重要な「個人の消費エネルギー（TDEE）に基づいた栄養管理と献立生成が十分に連携されていないため、理想の体型管理を仕組みとして完結させることができません。

<br>

#### 継続できない本質的な原因<br>
- ダイエットが続かない本質的な原因は、ユーザーの知識不足ではなく「意思決定の積み重ねによる疲弊」にあると考えています。<br>私は、情報を与えるだけでなく、行動を直接ナビゲートすることで、誰もが迷わずに健康的な食生活を継続できる仕組みを作りたいと考えています。

<br>

### 5-3. このアプリは、【どんな人】の【どんな瞬間】を一番助けるアプリか？

1日の摂取カロリーを意識しながら食事をしたい人の、「献立を考えるのが面倒な瞬間」を解決するアプリです。

## 7. このアプリで実現すること

### MVPで作る機能
1. 消費カロリー算出機能(TDEE診断)<br>
身体情報や運動量を入力することで、1日の消費カロリーを即座に計算。<br><br>
2. ユーザー認証・ゲストログイン機能<br>
メールアドレスによる新規登録、ログイン、パスワード再設定に加え、「ゲストログイン」を実装。<br>アカウント作成の手間なく、すべての機能をすぐに試せる環境を提供する。

3. 食物アレルギー設定機能（任意）<br>
ユーザーは食物アレルギーを選択式で登録可能。<br>
登録内容をもとに、該当食材を除外した献立を生成する。

4. 献立自動生成アルゴリズム<br>
目標カロリーと時間帯（朝・昼・晩）の属性に基づき、独自の献立データから条件に合う1日分（3食）の献立を選出。<br>
※本リリースにて、3日・4日・7日など生活スタイルに合わせた期間カスタマイズ機能を拡張予定。<br>
※MVPでは「何を食べるかを決める負担の軽減」にフォーカスするため、レシピ詳細機能は実装せず、献立提案に機能を絞る。

5. 指定日の献立閲覧機能<br>
生成された1日分（3食）の献立・合計カロリーを表示。<br>
ユーザーの生活リズムに合わせて、当日や翌日の献立を迷わず把握できる確認画面。


### 本リリースで作る機能
1. 献立自動生成アルゴリズム（期間拡張 / 生成AI連携）<br>
生成AI（LLM）と連携し、より柔軟で多様な献立提案を実現。

2. レシピ表示機能<br>
各献立に対応したレシピ・調理手順・必要食材を表示。<br>
「献立を決める」だけでなく、「実際に作る」までを一貫してサポートできるようにする。

3. 献立生成期間のカスタマイズ機能<br>
3日・4日・7日など、買い物頻度や生活スタイルに合わせて生成スパンを柔軟に設定可能にする。

4. 1週間献立表示機能（カレンダー形式）<br>
生成された献立をカレンダー形式で一覧表示。<br>中長期的な食事計画を視覚的に管理できる。

5. 献立の入れ替え（部分再生成）機能<br>
提案されたメニューの中で、特定の1食をワンタップで別の候補に入れ替える。

6. 食事実績の調整（ズレ補正）機能<br>
外食などで計画と異なる食事をした際、その内容を入力することで、残りの食事のカロリーバランスを自動で再計算・調整する。

7. お気に入り保存機能<br>
気に入った献立をストックし、次回の生成時に優先的に組み込めるようにする。

8. TDEE再計算リマインド・自動更新機能<br>
一定期間ごとに体重等の見直しを促し、常に最新の身体状態に合わせた献立を提案できるようにする。

### 将来的に作る機能
- 外部レシピAPIとの連携
膨大なレシピデータから、より信頼性の高い調理手順や画像を提供し、サービスの完成度を極める。

## 8. このアプリの懸念点とその対策
### 懸念点
ユーザーが「毎日完璧にこなさなければならない」というプレッシャーを感じて挫折してしまう

- 何が問題になりそうか？<br>献立を提案されても、ユーザーが「毎日この通りに作らなければならない」というプレッシャーを感じ、少しでもズレると「もういいや」と諦めてしまう可能性がある。

- なぜそう思うか<br>ダイエットは「完璧主義」に陥りやすく、少しでも計画通りにいかないと挫折しやすい傾向がある。

- そのために考えている対策は献立の入れ替え機能を追加することです。<br>
ユーザーが「今日はこのメニューは作れない」と感じたとき、ワンタップで別の献立に変更できるようにする。<br>※「気に入らない場合は変更できる」という柔軟性を持たせる。
```
具体例
【献立画面】

火曜日:鶏胸肉のソテー(調理時間30分)
「今日は無理…」と感じたとき
↓
【入れ替えボタンをタップ】
 豚肉の生姜焼き(調理時間15分)に変更
```

## 9. 今後の展開・発展の方向性
### 観点

- ユーザーはなぜ使い続ける？<br>
「何を食べるか」を考える必要がなくなることで、日々の意思決定の負担を減らし続けられるため。

- 続けることで何が変わる？<br>
食事管理を“頑張るもの”ではなく、“自然に続く習慣”として維持できるようになる。

- データが溜まると何ができる？<br>
ユーザーの好み・変更傾向・継続パターンをもとに、より自分に合った献立提案が可能になる。

### 今考えている発展案

- 機能追加の方向性：パーソナライズ献立機能<br>
ユーザーが「よく選ぶ献立」「入れ替えたメニュー」「継続しやすかった食事傾向」を学習し、より好みに合った献立を優先的に提案できるようにする。

- UI・体験の改善：柔軟性を感じられる操作体験<br>
入力や設定を最小限にし、「ボタン1つで今日の献立が決まる」状態を目指す。
また、献立の再生成や変更も直感的に行えるUIへ改善する。

- 機能以外の工夫：季節・旬の食材アップデート<br>
季節やトレンドに合わせてレシピデータを定期更新し、「飽きにくさ」や「作りやすさ」を維持する。
また、調理時間や食材コストなど、現実的に続けやすい要素も重視した提案を行う。

| Devise              | ユーザー認証（新規登録/ログイン/パスワード再設定）を1ファイルの設定で実装可能。ゲストログインも標準対応しており、MVPの「60〜120秒で献立生成開始」というUX要件を確実に満たし、セキュリティもプロフェッショナルレベルで担保。                      |
| ActiveRecord        | Rails標準ORMで、食材タグ検索（where, joins, includes）、TDEE計算結果の永続化、献立履歴管理を直感的に実装可能。PostgreSQLとの相乗効果で、複雑な条件フィルタリング（アレルギー除外、カロリー範囲、食事タイプ別抽出）も1クエリで効率化。 |
| Active Storage      | レシピ写真登録・表示をS3互換ストレージでゼロコード実装。管理画面での食材写真登録や、将来的なユーザーアバター、お気に入り献立のサムネイル表示まで一貫して対応し、画像関連の開発工数を大幅削減。                                          |

## インフラ・その他
| 技術             | 選定理由                            |
| -------------- | ----------------------------------- |
| PostgreSQL     | Railsとの相性が良く、信頼性・拡張性が高いため。          |
| Render         | Rails 7.2環境をデプロイしやすく、コストを抑えた運用が可能なため。      
| Docker         | 開発環境を統一し、環境差異によるトラブルを防ぐため。          |
| Node.js / Yarn | Tailwind CSSなどのビルドツールを利用するため。       |
| ActiveJob      | Rails標準のジョブ機能として、リマインドや週次処理を実装するため。 |
| Sidekiq　　     | ジョブ処理のパフォーマンスを向上させるため（本リリースで検討）。    |
| RSpec          | ドキュメントとしても読めるテストが書け、保守性を高められるため。    |
| Resend         | 認証メール（パスワード再設定など）を安定して送信するため。       |
| whenever | cronベースで定期実行を簡単に設定できるため |
